如果你已经在用 tmux，但又不想从零配一套 `.tmux.conf`，那 `Oh My Tmux` 基本就是最省事的一条路。

这篇不是官方 README 的整段搬运，而是按自己的使用习惯做的整理版。原项目在这里：

- [gpakosz/.tmux](https://github.com/gpakosz/.tmux)

## 先说结论

`Oh My Tmux` 适合这种人：

- 已经在用 tmux
- 想要一套成熟、顺手、可继续深改的默认配置
- 不想从头调状态栏、按键、剪贴板、鼠标模式和远程行为

它的核心优点不是“花”，而是：

- 默认值已经很能打
- 定制入口清楚
- 文档足够完整
- 长期维护稳定

## 它到底是什么

官方对它的定位很直接：这是一个 **self-contained、pretty、versatile** 的 tmux 配置。

你可以把它理解成：

- 不是 tmux 的替代品
- 也不是单纯的一组主题
- 而是一套完整的 tmux 默认工作流

它直接帮你处理了这些高频问题：

- 前缀键怎么定
- 状态栏怎么配
- 复制粘贴怎么顺手
- 鼠标模式怎么切
- pane / window / session 怎么更高效切换
- 本地和 SSH / Mosh 场景下的行为怎么统一

## 安装前要知道的事

README 里有几条我认为必须先记住：

### 1. 依赖

至少要满足：

- tmux `>= 2.6`
- Linux / macOS / OpenBSD / Windows（Windows 更推荐 WSL）
- `awk`、`perl`、`grep`、`sed`
- tmux 外部的 `TERM` 环境变量建议设成 `xterm-256color`

### 2. 不要改主配置文件

这一条是整个项目最重要的使用约定：

- 不要直接改 `.tmux.conf`
- 真正的自定义应该写在 `.tmux.conf.local`

这是它最对的一点。  
这样做的好处非常实际：

- 主仓库更新时不容易冲掉你的改动
- 你自己的定制和上游默认配置分层清楚
- 出问题时更容易定位是“你改坏了”还是“上游配置行为变化”

如果你要长期用，最好从第一天就遵守这条。

## 安装方式

### 自动安装

README 提供了自动安装脚本：

```bash
curl -fsSL "https://github.com/gpakosz/.tmux/raw/refs/heads/master/install.sh#$(date +%s)" | bash
```

适合你只是想快速起步。

### 手动安装

如果想更明确地知道发生了什么，我更推荐手动方式。

最常见的是装到 `~`：

```bash
cd ~
git clone --single-branch https://github.com/gpakosz/.tmux.git
ln -s -f .tmux/.tmux.conf
cp .tmux/.tmux.conf.local .
```

如果你走 XDG 目录，也可以装到：

- `$XDG_CONFIG_HOME/tmux`
- `~/.config/tmux`

这一点也挺好：它没有把目录结构写死。

---

## 初次安装后的第一件事

不是改 `.tmux.conf`，而是打开：

```bash
~/.tmux.conf.local
```

官方甚至专门给了快捷键，装好以后你可以直接用：

- `prefix + e`

来打开本地自定义配置文件。

这就是它的正确用法：

1. 保持主配置不动
2. 只在 `.tmux.conf.local` 里做改动
3. 改完直接 reload

## 它默认做了哪些事

我觉得最值钱的是下面这些。

### 1. `C-a` 作为二级前缀

它保留了 tmux 默认的 `C-b`，同时再给你一个 `C-a` 作为 secondary prefix。

这对很多人是更顺手的，因为：

- 手感接近 GNU Screen
- 某些键位组合更自然

但关键在于：它没有粗暴覆盖默认 `C-b`，所以迁移成本低。

### 2. Powerline 风格状态栏

这是大家最容易注意到的部分。

它默认给你的是一套比较成熟的 Powerline 风格状态栏，支持：

- 电池信息
- uptime
- 当前用户和主机名
- SSH / Mosh 感知
- prefix / pairing / synchronized 等状态

如果你喜欢状态栏可读性强的终端工作流，这套默认值确实很省时间。

### 3. 更合理的 pane / window 操作

它把很多高频操作改得更顺手，比如：

- Vim 风格 pane 导航
- Vim 风格 pane resize
- 更直接的 split 绑定
- 最大化 pane 到新窗口

这类体验上的改进，长期用下来比主题更值钱。

### 4. 更照顾远程场景

它对：

- SSH
- Mosh
- 远程分屏
- 远程主机名 / 用户名展示

这些场景考虑得比较到位。  
如果你经常在跳板机、远程开发机和本地机之间切换，这一点会很明显。

## 我觉得最值得记的快捷键

这里不列全表，只记高频的。

前提：下面的 `prefix` 指的是 `Ctrl+a` 或 `Ctrl+b`。

### 配置相关

- `prefix + e`
  打开 `.tmux.conf.local`
- `prefix + r`
  重载配置

### session / window

- `prefix + Ctrl-c`
  新建 session
- `prefix + Ctrl-f`
  按名称切 session
- `prefix + Tab`
  回到上一个活动 window
- `prefix + Ctrl-h`
- `prefix + Ctrl-l`
  在 window 之间切换

### pane

- `prefix + -`
  垂直分屏
- `prefix + _`
  水平分屏
- `prefix + h / j / k / l`
  Vim 风格 pane 导航
- `prefix + H / J / K / L`
  调整 pane 尺寸
- `prefix + < / >`
  交换 pane
- `prefix + +`
  最大化当前 pane 到新窗口

这里的 `+` 是它的一个很实用功能。它和 tmux 原生 `resize-pane -Z` 不完全一样。README 里也专门说明了：它允许把 pane 独立拉到一个新窗口里，再回来时仍保留这层关系。

### 其他高频

- `prefix + m`
  开关鼠标模式
- `prefix + Enter`
  进入 copy-mode
- `prefix + b`
  列 paste buffers
- `prefix + p`
  粘贴顶部 buffer
- `prefix + P`
  选择某个 buffer 再粘贴

## copy-mode-vi 很顺手

如果你本来就是 Vim 用户，这套默认配置很容易上手。

README 里提到的高频操作包括：

- `v`
  开始选择
- `Ctrl-v`
  在块选择和普通选择之间切换
- `H`
  跳到行首
- `L`
  跳到行尾
- `y`
  复制选择内容
- `Escape`
  取消当前操作

如果你的 `VISUAL` 或 `EDITOR` 设成 `vim`，还会进一步启用和优化 vi-style bindings。

## Powerline 外观怎么开

有的人装完以后最先想问的就是：  
“为什么我的状态栏没有 README 截图里那么像 Powerline？”

原因通常不是配置没生效，而是字体。

README 里给的路线很清楚：

- 用自带 Powerline 符号的字体
- 或者用 patch 过的字体
- 或者单独安装 `PowerlineSymbols.otf`

然后在 `.tmux.conf.local` 里调这几个分隔符：

```bash
tmux_conf_theme_left_separator_main='\uE0B0'
tmux_conf_theme_left_separator_sub='\uE0B1'
tmux_conf_theme_right_separator_main='\uE0B2'
tmux_conf_theme_right_separator_sub='\uE0B3'
```

如果你不想折腾字体，也完全可以不启用这套符号。  
这点 README 也讲得很实在：**不是必须上 Powerline 才能正常用**。

## 状态栏定制怎么做

真正要调的不是源码，而是 `.tmux.conf.local` 里的变量。

最核心的是：

- `tmux_conf_theme_status_left`
- `tmux_conf_theme_status_right`

它已经内置了很多可用变量，比如：

- `#{battery_percentage}`
- `#{battery_status}`
- `#{hostname}`
- `#{username}`
- `#{loadavg}`
- `#{uptime_d}`
- `#{uptime_h}`
- `#{prefix}`
- `#{root}`
- `#{synchronized}`

也支持标准 tmux 的 `#()`，所以你可以塞自己的 shell 命令进去。

比如 README 里就给了用 `wttr.in` 拉天气的例子。

我的建议是：

- 先少改
- 只放你真的会看的信息
- 状态栏别做成信息垃圾场

## TPM 插件怎么配

`Oh My Tmux` 自带 TPM 支持，这一点很省事。

你要做的是：

- 用 `set -g @plugin ...` 开插件

而不要做的是：

- 不要手动再加 `tmux-plugins/tpm`
- 不要自己再写 `run '~/.tmux/plugins/tpm/tpm'`

这点如果重复配置，很容易把行为弄乱。

另外它的 TPM 快捷键和原版略有区别：

- 安装插件：`prefix + I`
- 卸载插件：`prefix + Alt-u`
- 更新插件：`prefix + u`

## Windows 怎么用

README 对 Windows 的态度很明确：

- **不再推荐 Cygwin**
- 推荐 **WSL + Windows Terminal**

原因也很现实：

- Cygwin 下 forking 很慢
- 这套配置内部会频繁跑 `run-shell`
- 结果就是 CPU 高、体验差

所以如果你在 Windows 上认真用 tmux，这个项目给出的路线其实和现在大多数人的经验一致：

- 终端：Windows Terminal
- tmux 环境：WSL

这也是我会推荐的用法。

## 排错时别直接硬怼配置

README 给了一个很重要的基准测试方法：

```bash
tmux -f /dev/null -L test
```

这个命令的作用是：

- 起一个全新的 tmux client/server
- 不加载任何配置

它非常适合判断问题到底来自：

- tmux 本体
- 还是你当前加载的配置

如果你以后折腾状态栏、插件、字体、远程剪贴板时出问题，这个命令比盲猜高效得多。

## 我的使用建议

如果你准备长期用 `Oh My Tmux`，我建议这样上手：

1. 先手动安装，不要一上来就魔改
2. 主配置不动，只改 `.tmux.conf.local`
3. 先熟悉它的默认快捷键
4. 再逐步调状态栏和外观
5. 插件最后再加，不要起步就堆一排

原因很简单：  
它本身已经是一套成熟工作流。你应该先用它，再决定哪里真的要改。
